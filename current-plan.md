## Критично

### [ ] 1. Down-алерты подавляются активным SSL-инцидентом

`src/alerts.ts:112` — открытие down-инцидента требует `!monitor.active_incident_id`.
Если у монитора открыт warning-инцидент `ssl_expiry` («cert expires in N days») и сайт
падает — down-инцидент не открывается и **уведомление не отправляется**. Статус на
странице показывает Outage, но алерта нет.

**Что делать:** разрешить сосуществование инцидентов разных классов (connectivity vs
`ssl_expiry`) — либо два слота состояния, либо проверка «нет активного инцидента
*этого класса*» вместо глобального `active_incident_id`.

**Критерий приёмки:** монитор с открытым SSL-warning при падении открывает critical
down-инцидент и шлёт уведомление; при восстановлении оба инцидента живут своими
жизненными циклами.

### [ ] 2. Приватные мониторы не должны утекать при отключённом auth

`src/routes.ts:241` — если `auth_config` отсутствует или `enabled = 0`, то
`authEnabled = false` → `showAll = true` для всех: приватные мониторы и их
target-URL (внутренние адреса) становятся публичными. Плюс окно до 5 минут из-за
кэша auth-конфига (`src/auth.ts:5`).

**Что делать:** инвертировать default — `visibility = 'private'` скрывается всегда,
когда нет валидной сессии. Отсутствие auth-конфига = «только публичное», а не
«всё открыто». Сократить/инвалидировать кэш auth-конфига при включении auth.

**Критерий приёмки:** при пустой/выключенной `auth_config` запросы `/private`,
`/api/status`, `/api/history` без валидного JWT возвращают только public-мониторы
и не содержат `scrape_url`, алертов и usage-блока.

## Важно

### [ ] 4. Ретеншн: monitor_executions, incidents, notification_deliveries

Сейчас еженедельная чистка (`src/scheduler.ts:159`) удаляет только `metric_series`
и `uptime_hourly`. Лежащий монитор пишет ~1440 строк executions/день.

**Принятое решение (анализ от 2026-06-11):**
- `monitor_executions` — хранить **48 часов** (в UI не используется, только debug);
- `incidents` со статусом `resolved` — хранить **120 дней**: статус-страница читает
  инциденты на 90 дней назад (`src/routes.ts:185`, запрос `allIncidents` для
  раскраски баров и тултипов), 7-дневный ретеншн из ARCHITECTURE.md сломал бы UI;
- открытые инциденты **не удалять никогда**;
- `notification_deliveries` — отдельная чистка не нужна: FK
  `ON DELETE CASCADE` по `incident_id` удалит их вместе с инцидентами;
- чистку перевести с еженедельной на **ежедневную** (пропущенный тик стоит день,
  а не неделю); пока в рамках минутной арифметики в scheduled-обработчике,
  отдельные cron-выражения — в ROADMAP.

**Критерий приёмки:** после ежедневного запуска чистки в D1 нет executions старше
48ч и resolved-инцидентов старше 120 дней; 90-дневные бары и тултипы инцидентов
на странице не деградируют.

### [ ] 5. Retry уведомлений

`src/queue.ts:42` — сообщение ack-ается независимо от результата доставки,
`notification_deliveries.attempt_count` всегда 1.

**Что делать:** при неуспехе доставки во все каналы — `msg.retry()` (у Queues
встроенный retry с backoff, настроить `max_retries` в `wrangler.jsonc`);
фиксировать реальный `attempt_count` в `notification_deliveries`. Каналы
`email`/`telegram` не реализованы — убрать email-канал из `config.yaml` или
явно пометить нереализованным, чтобы алерты не уходили молча в `failed`.

**Критерий приёмки:** временный сбой вебхука (5xx) приводит к повторной доставке;
после исчерпания ретраев delivery помечен `failed` с реальным числом попыток.

### [ ] 6. Планировщик: голодание при >15 due-мониторов

`src/scheduler.ts:105` — `slice(0, MAX_CHECKS_PER_RUN)` без сортировки: при числе
due-мониторов больше 15 в срез стабильно попадают одни и те же, остальные не
проверяются никогда.

**Что делать:** сортировать due-мониторы по давности проверки
(`last_check_at ASC NULLS FIRST`) перед срезом.

**Критерий приёмки:** при 20+ due-мониторах каждый проверяется не реже, чем раз
в `ceil(N / 15)` минут.

### [ ] 7. Атомарность открытия/закрытия инцидентов и счётчиков

`src/alerts.ts:122-137` — INSERT инцидента, UPDATE `monitor_state` и отправка в
очередь — три отдельные операции: падение между ними даёт инцидент-сироту
(следующий тик откроет дубль) или инцидент без уведомления. Счётчики
`consecutive_*` считаются в JS от снапшота — параллельные записи теряют инкременты.

**Что делать:** INSERT инцидента + UPDATE state завернуть в `env.DB.batch()`
(то же для resolve); инкременты счётчиков перенести в SQL
(`SET consecutive_failures = consecutive_failures + 1`); отправку в очередь
выполнять после успешного батча.

**Критерий приёмки:** нет пути исполнения, при котором инцидент открыт, а
`active_incident_id` не выставлен (и наоборот).

### [ ] 9. Edge-кэш для /public

`src/routes.ts:145` — публичная страница отдаётся с `Cache-Control: no-store` и
кэшируется только в памяти изолята: каждый запрос тратит 1 из 100k requests/day.
Всплеск трафика на статус-пейдж приходит ровно во время инцидента.

**Что делать:** на `/public` (и публичный `/api/history`?) отдавать
`Cache-Control: public, max-age=60`, чтобы edge-кэш гасил повторные запросы без
вызова Worker и чтения D1. In-memory кэш изолята можно оставить как второй слой.

**Критерий приёмки:** повторный запрос `/public` в течение TTL отдаётся из edge
(`cf-cache-status: HIT`), Worker не вызывается.

## Отложено до выхода из беты

### [ ] 11. import-config: INSERT OR REPLACE каскадно стирает runtime-данные

`scripts/import-config.ts:168` — `INSERT OR REPLACE INTO monitors` = DELETE +
INSERT; через `ON DELETE CASCADE` на каждом деплое стираются `monitor_state`,
`alert_rules`, `incidents`, `monitor_executions`, `metric_series` всех мониторов
из конфига. Выживают только `uptime_*` (без FK), поэтому бары выглядят целыми.

**Решение принято 2026-06-11:** пока проект в активной разработке, потеря
runtime-состояния на деплое приемлема. Зафиксировать схему БД и заменить
`INSERT OR REPLACE` на `INSERT ... ON CONFLICT(id) DO UPDATE SET ...`
(monitors, alert_rules, notification_channels) при выходе из беты.
